# 技術的課題と将来の拡張計画 — BPO業務進捗チェックシステム

> 作成日: 2026-03-05
> 対象プロジェクト: `timezone-task-app`（Next.js 16 + React 19 + TypeScript）

---

## 1. 概要

本ドキュメントは、現在の実装における技術的な課題を整理し、将来の機能拡張・品質改善に向けたアドバイスをまとめたものである。

## 2. 現在の実装課題（優先度：高）

### 2-1. 認証情報のセキュリティリスク

**現状:**
`NEXT_PUBLIC_*` プレフィックスの環境変数は、Next.js ビルド時にクライアントサイドの JavaScript バンドルに**平文で埋め込まれる**。ブラウザの DevTools でソースコードを確認すれば誰でも読み取ることができる。

```bash
# ブラウザでバンドルを確認すると認証情報が見える
NEXT_PUBLIC_ADMIN_ID=admin
NEXT_PUBLIC_ADMIN_PASSWORD=admin
```

**リスク:**
- 現在のパスワードは既に公知（`user`/`user`, `admin`/`admin`）であり、社外秘データへのアクセス制御として機能していない
- 将来パスワードを変更しても、バンドルを再ビルドするまで古い認証情報が残る可能性がある

**推奨対応:**
- サーバーサイドで認証を行う API Route (`app/api/login/route.ts`) を実装する
- または NextAuth.js / Clerk などの認証ライブラリを導入する
- 暫定対応として、認証情報をサーバー専用環境変数（`NEXT_PUBLIC_` なし）に移行する

```
# サーバー専用（クライアントに露出しない）
ADMIN_ID=admin
ADMIN_PASSWORD=<strong-password>
```

### 2-2. localStorage によるデータ永続化の限界

**現状:**
全ての業務データは各ユーザーのブラウザの `localStorage` に保存される。キー形式: `task-checker-v3-YYYY-MM-DD`。

**課題:**

| 課題 | 詳細 |
|-----|------|
| デバイス間共有不可 | 同じ担当者が別のPCや端末でアクセスしても、データが引き継がれない |
| リアルタイム同期なし | 管理者が管理画面を表示しても、作業者のリアルタイム進捗は見えない（手動リロード必要） |
| 容量制限 | localStorage の上限は約 5MB。多数の日付データが蓄積すると書き込み失敗のリスクがある |
| データ損失リスク | ブラウザのキャッシュクリア・プライベートブラウジング・端末故障でデータが消える |
| バックアップ不可 | サーバーサイドにデータが存在しないため、管理者が過去データを取得できない |

---

### 2-3. sessionStorage による認証の脆弱性

**現状:**
ログイン成功時に `sessionStorage.setItem('auth-user', 'true')` を保存し、存在確認でアクセス制御を行っている。

**課題:**
- ブラウザの DevTools Console から `sessionStorage.setItem('auth-admin', 'true')` を実行するだけで管理者画面にアクセスできてしまう
- セッションの有効期限管理がない（タブを閉じるまで有効）
- 同一ブラウザ内の別タブから sessionStorage を参照できる（タブ間の分離は保証されない）

## 3. 現在の実装課題（優先度：中）

### 3-1. task-checker.tsx の肥大化

**現状:** [components/task-checker.tsx](../components/task-checker.tsx) は 675 行。

**課題:**
1 ファイルに以下の責務が混在している。
- 地域選択 UI
- タスク一覧テーブル
- カスタムタスク追加フォーム
- 業務日報セクション（`ReportSection` コンポーネントは分離済み）
- localStorage からのデータロード・保存ロジック

**推奨対応:**
```
components/
├── task-checker/
│   ├── index.tsx           # メインコンポーネント（状態管理のみ）
│   ├── RegionSelector.tsx  # 地域選択グリッド
│   ├── TaskTable.tsx        # タスク一覧テーブル
│   ├── CustomTaskForm.tsx  # カスタムタスク追加フォーム
│   └── ReportSection.tsx   # 業務日報（既に分離済み）
```

### 3-2. admin-dashboard.tsx の `mergedSlots()` が非メモ化

**現状:** [components/admin-dashboard.tsx](../components/admin-dashboard.tsx) の `mergedSlots()` 関数はレンダリングのたびに再計算される。

```typescript
// 現状（毎レンダリング再計算）
const mergedSlots = () => getMergedTimeSlots([])
```

**課題:**
管理画面では全 7 地域のデータを表示するため、`mergedSlots()` が頻繁に呼ばれると不要な再計算が発生する。

**推奨対応:**
```typescript
// useMemo でメモ化
const mergedSlots = useMemo(() => getMergedTimeSlots([]), [])
```

### 3-3. エラー発生時のユーザーフィードバックがない

**現状:**
localStorage への書き込み失敗（容量超過など）は `console.warn` でログを出すだけで、**ユーザーには何も通知されない**。

**推奨対応:**
`sonner`（既にインストール済み）を使ったトースト通知で警告を表示する。

```typescript
import { toast } from 'sonner'

// saveDayData でエラー発生時
toast.error("データの保存に失敗しました。ブラウザのストレージ容量を確認してください。")
```

### 3-4. タイムゾーンが JST 固定

**現状:**
`lib/task-utils.ts` の `getDateKey()` は `new Date()` の JST 表示に依存している。

```typescript
// JST 環境外では日付がずれる可能性
export function getDateKey(date?: Date): string {
  const d = date ?? new Date()
  return `${d.getFullYear()}-${...}-${...}`
}
```

**課題:**
Vercel のサーバーレンダリング時（UTC 環境）と クライアントサイド（JST 環境）で日付が 9 時間ずれる可能性がある。現在はクライアントサイドのみで実行されているため問題ないが、将来サーバーサイドで使用する場合は注意が必要。

**推奨対応:**
```typescript
// Intl.DateTimeFormat で JST を明示
const jstDate = new Date().toLocaleDateString('ja-JP', {
  timeZone: 'Asia/Tokyo',
  year: 'numeric', month: '2-digit', day: '2-digit'
})
```

## 4. 将来の拡張アドバイス

### 4-1. バックエンド追加（最優先推奨）

**目的:** 複数端末間でのデータ共有・リアルタイム同期の実現

**推奨オプション（Vercel プロジェクトの場合）:**

| オプション | 特徴 | 難易度 |
|---------|-----|-------|
| **Vercel KV（Upstash Redis）** | Vercel との統合が容易、無料枠あり | ★★☆ |
| **Supabase** | PostgreSQL + リアルタイム購読、OSS | ★★★ |
| **Prisma + Neon** | TypeScript 親和性が高い PostgreSQL | ★★★ |

**データスキーマイメージ（Supabase の場合）:**
```sql
-- 日次データ
CREATE TABLE day_data (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  date date NOT NULL,
  region_id text NOT NULL,
  staff_name text,
  completed_tasks jsonb DEFAULT '[]',
  custom_tasks jsonb DEFAULT '[]',
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

-- 業務日報
CREATE TABLE daily_reports (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  date date NOT NULL,
  new_items jsonb DEFAULT '[]',
  shared_items jsonb DEFAULT '[]'
);
```

### 4-2. 認証強化

**推奨ライブラリ:**

| ライブラリ | 特徴 | 推奨シナリオ |
|---------|-----|-----------|
| **NextAuth.js (Auth.js)** | Next.js 標準、Google/GitHub/Credentials 対応 | 社内ツールの簡易認証 |
| **Clerk** | UI コンポーネント付き、SaaS 向け | 多機能・多テナント |
| **Supabase Auth** | バックエンドと一体化 | Supabase を採用する場合 |

**最小構成（NextAuth.js + Credentials）:**
```typescript
// app/api/auth/[...nextauth]/route.ts
import NextAuth from 'next-auth'
import Credentials from 'next-auth/providers/credentials'

export const { handlers } = NextAuth({
  providers: [
    Credentials({
      authorize: async (credentials) => {
        // サーバーサイドで認証情報を検証（DB照合など）
        if (credentials.id === process.env.ADMIN_ID) {
          return { id: '1', role: 'admin' }
        }
        return null
      }
    })
  ]
})
```

### 4-3. リアルタイム進捗共有

**目的:** 管理者が全地域の進捗をリアルタイムで確認できる

**推奨実装:** Vercel KV + Server-Sent Events（SSE）

```typescript
// app/api/progress/route.ts
export async function GET() {
  const encoder = new TextEncoder()
  const stream = new ReadableStream({
    async start(controller) {
      // 進捗データを定期的にプッシュ
      const interval = setInterval(async () => {
        const data = await kv.get('progress')
        controller.enqueue(encoder.encode(`data: ${JSON.stringify(data)}\n\n`))
      }, 5000)
    }
  })
  return new Response(stream, {
    headers: { 'Content-Type': 'text/event-stream' }
  })
}
```

## 5. コード品質の改善アドバイス

### 5-1. 型安全性の強化

**現状:** `DaySaveData` 型のバリデーションがない。`JSON.parse` 後のデータが型定義に合致するか確認されていない。

**推奨:** `zod` を使ったランタイムバリデーションの追加。

```typescript
// lib/task-utils.ts に追加
import { z } from 'zod'

const CompletedTaskSchema = z.object({
  taskId: z.string(),
  regionId: z.string(),
  completedAt: z.string(),
})

const DaySaveDataSchema = z.object({
  completedTasks: z.array(CompletedTaskSchema),
  report: z.object({
    newItems: z.array(ReportItemSchema),
    sharedItems: z.array(ReportItemSchema),
  }),
  // ...
})

// loadDayData 内でバリデーション
const parsed = DaySaveDataSchema.safeParse(JSON.parse(stored))
if (!parsed.success) return emptyDayData()
return parsed.data
```

## 6. 運用上の注意事項

| 項目 | 詳細 |
|-----|------|
| `.env.local` の管理 | `.gitignore` に含まれているが、Vercel デプロイ時は Dashboard の Environment Variables に設定が必要 |
| 認証情報の変更 | 本番環境では `NEXT_PUBLIC_*` の値を必ず強固なパスワードに変更すること（デフォルト値は `user`/`admin` で誰でも分かる） |
| localStorage の自動クリーンアップ | `cleanupOldData()` が起動時に実行され、2ヶ月超のデータを自動削除する。重要なデータは CSV ダウンロードで保管すること |
| Playwright テストの実行環境 | WSL2 では Chromium の依存ライブラリが不足するため、以下の設定が必要：`LD_LIBRARY_PATH=/tmp/lib-extract/usr/lib/x86_64-linux-gnu` |
| next-themes の互換性 | React 19 + Next.js 16 環境での next-themes は `suppressHydrationWarning` が必須。削除しないこと |
