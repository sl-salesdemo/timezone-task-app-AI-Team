## 2. Phase A-2: ThemeProvider ハイドレーションバグ修正

### 問題

`next-themes` を使用したダークモード機能において、ページ初回表示時に「ライトモード → ダークモード」が一瞬切り替わる **FOUC（Flash of Unstyled Content）** が発生していた。

原因は `app/layout.tsx` に `ThemeProvider` のラップと `suppressHydrationWarning` の設定が欠落していたこと。

### 対応内容

**変更ファイル:** [app/layout.tsx](../app/layout.tsx)

```tsx
// Before（問題あり）
<html lang="ja">
  <body>
    {children}
    <Analytics />
  </body>
</html>

// After（修正済み）
<html lang="ja" suppressHydrationWarning>
  <body>
    <ThemeProvider attribute="class" defaultTheme="system" enableSystem>
      {children}
      <Analytics />
    </ThemeProvider>
  </body>
</html>
```

**修正ポイント:**
- `<html>` タグに `suppressHydrationWarning` を追加 → テーマ切り替えによる hydration ミスマッチを抑制
- `ThemeProvider` で `{children}` をラップ → 全コンポーネントにテーマコンテキストを提供
- `defaultTheme="system"` → OS のダーク/ライト設定に自動追従

## 3. Phase B: 共通ユーティリティ化（lib/task-utils.ts）

### 問題

`components/task-checker.tsx`（776行）と `components/admin-dashboard.tsx`（402行）の両コンポーネントに以下の重複が存在していた。

| 重複箇所 | 内容 |
|---------|------|
| 型定義 | `CompletedTask`, `DailyReport`, `ReportItem`, `RegionAssignment`, `CustomTask`, `DaySaveData` |
| localStorage 操作 | `getStorageKey`, `loadDayData`, `saveDayData`, `cleanupOldData` |
| ビジネスロジック | `isLate`（遅延判定）, `getMergedTimeSlots`（タスクマージ） |

重複があるとバグ修正時に両方の変更が必要となり、修正漏れのリスクが高い。

### 対応内容

**新規作成ファイル:** [lib/task-utils.ts](../lib/task-utils.ts)

共通の型定義と関数を 1 ファイルに集約し、両コンポーネントからインポートする構成に変更した。

**抽出した主要関数:**

| 関数名 | 説明 | エラーハンドリング |
|-------|------|-----------------|
| `getDateKey(date?)` | YYYY-MM-DD 形式の日付キー取得 | — |
| `getStorageKey(dateKey?)` | `task-checker-v3-{date}` 形式のキー生成 | — |
| `loadDayData(dateKey?)` | localStorage からデータ読み込み | JSON.parse 失敗時にデフォルト値を返す |
| `saveDayData(data, dateKey?)` | localStorage にデータ書き込み | QuotaExceededError を catch して警告出力 |
| `cleanupOldData()` | 2ヶ月超の古いデータを自動削除 | 例外を catch して処理継続 |
| `getAvailableDates()` | 過去2ヶ月の日付リストを降順で返す | — |
| `getMergedTimeSlots(customTasks)` | デフォルト + カスタムタスクを時刻順マージ | — |
| `isLateCalc(slotTime, completedAt)` | 完了時刻が理想時間+30分超か判定 | 不正入力時は false を返す |

**エラーハンドリング強化:**
既存の localStorage 操作にはエラーハンドリングがなく、JSON.parse 失敗や容量超過でページがクラッシュするリスクがあった。全ての localStorage 操作を `try-catch` で囲み、安全にフォールバックする実装に変更した。

```typescript
// 修正前（クラッシュリスクあり）
const stored = localStorage.getItem(key)
return JSON.parse(stored) // stored が null や不正 JSON の場合に例外

// 修正後（安全）
try {
  const stored = localStorage.getItem(key)
  if (!stored) return emptyDayData()
  return JSON.parse(stored) as DaySaveData
} catch (err) {
  console.warn("[task-utils] loadDayData: 読み込み失敗、デフォルト値を使用します", err)
  return emptyDayData()
}
```

**変更ファイル:**
- [components/task-checker.tsx](../components/task-checker.tsx) — 重複型・関数を削除し `lib/task-utils` からインポート
- [components/admin-dashboard.tsx](../components/admin-dashboard.tsx) — 同上

### C-1: 認証情報の環境変数化

#### 問題

`components/login-form.tsx` に認証情報がハードコードされていた。

```typescript

const CREDENTIALS = {
  user: { id: "user", password: "user" },
  admin: { id: "admin", password: "admin" },
}
```

ソースコードに認証情報が含まれると、GitHub 等への誤公開や将来の認証情報変更が困難になる。

```

NEXT_PUBLIC_USER_PASSWORD=user
NEXT_PUBLIC_ADMIN_ID=admin
NEXT_PUBLIC_ADMIN_PASSWORD=admin
```

**変更ファイル:** [components/login-form.tsx](../components/login-form.tsx)

```typescript
// 修正後（環境変数 + フォールバック）
const CREDENTIALS = {
  user: {
    id: process.env.NEXT_PUBLIC_USER_ID ?? "user",
    password: process.env.NEXT_PUBLIC_USER_PASSWORD ?? "user",

    id: process.env.NEXT_PUBLIC_ADMIN_ID ?? "admin",

```

## 6. テスト中に発見・修正したバグ

テスト自動化の過程で、以下の 4 件のバグを発見・修正した。

### BUG-001: button locator の曖昧マッチによる誤クリック

**発見テスト:** TC-05-1, TC-05-3, TC-05-4, TC-06-2, TC-07-6, TC-08

**問題:**
`page.locator('button', { hasText: '追加' }).first()` というセレクターが、DOM 上で先に存在する「**タスク追加**」ボタンにマッチしていた。その結果、業務日報への追加ボタンが実際には押されず、アイテムが追加されなかった。

**修正:**
```typescript
// 修正前（曖昧マッチ）
await page.locator('button', { hasText: '追加' }).first().click()

// 修正後（完全一致）
await page.getByRole('button', { name: '追加', exact: true }).click()
```

**影響ファイル:** e2e/report.spec.ts, e2e/persistence.spec.ts, e2e/admin.spec.ts, e2e/csv.spec.ts

---

### BUG-002: `text=理想時間` の strict mode violation

**発見テスト:** TC-02-4

**問題:**
`page.locator('text=理想時間')` が 2 要素にマッチした。
1. テーブルヘッダー `<th>理想時間</th>`
2. 凡例テキスト `<span>黄色マーカー：理想時間より30分以上遅く完了</span>`

Playwright の strict mode では複数マッチが禁止されているため、テストが失敗した。

**修正:**
```typescript
// 修正前
await expect(page.locator('text=理想時間')).toBeVisible()

// 修正後（ロールで特定）
await expect(page.getByRole('columnheader', { name: '理想時間' })).toBeVisible()
```

**影響ファイル:** e2e/region-staff.spec.ts

---

### BUG-003: CSV 出力でサブタスクの → プレフィックスが欠落

**発見テスト:** TC-08-6

**問題:**
UI 上ではサブタスクに `→ ` プレフィックスを付けて表示していたが（task-checker.tsx:600）、CSV 出力では `task.label` をそのまま使用していたため、サブタスクであることが CSV で識別できなかった。

**修正:** [components/admin-dashboard.tsx](../components/admin-dashboard.tsx) の CSV 生成部分を修正。

```typescript
// 修正前
csvContent += `${region.name},${staff},${slot.time},${task.label},...`

// 修正後
const taskLabel = `${task.isSubTask ? "→ " : ""}${task.label}`
csvContent += `${region.name},${staff},${slot.time},${taskLabel},...`
```

**影響ファイル:** components/admin-dashboard.tsx

---

### BUG-004: localStorage への自動保存完了前の画面遷移

**発見テスト:** TC-07-6, TC-08

**問題:**
React の `useEffect` によるデータの自動保存は非同期で実行される。タスクを完了させた直後に管理画面へ遷移すると、localStorage への書き込みが間に合わず、管理画面でデータが空に見えることがあった。

**修正:** localStorage へのキー書き込みを待機する `waitForFunction` を追加。

```typescript
// 追加した待機処理
await page.waitForFunction(() =>
  Object.keys(localStorage).some(k => k.startsWith('task-checker-v3-'))
)
```

**影響ファイル:** e2e/admin.spec.ts, e2e/csv.spec.ts
