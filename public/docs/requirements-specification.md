## 1. プロジェクトの概要

### 1.1 背景

本プロジェクトは、「BPO業務における日次タスクの進捗管理と業務報告をデジタル化し、地域別の業務効率化と品質向上を図る、業務進捗チェックシステムの実現」を目的として開発・運用されるシステムである。

その背景としては、BPO業務に従事するスタッフが、日々の定型業務（e-windsデータ処理、欠品振替、出荷確定、フラグ操作等）を各地域（関東、関西、九州、東海、北海道、東北、沖縄）ごとに遂行しており、その進捗管理が紙やスプレッドシートで行われている現状がある。この従来方式では以下の課題が生じている。

- 各地域の担当者間でタスク進捗のリアルタイムな可視化ができない
- 理想時間に対する遅延の検知と記録が手動で煩雑である
- 日次業務日報の作成・共有が非効率である
- 過去の実績データの蓄積と参照が困難である
- CSVレポートの出力に都度の手作業が必要である

そこで、本システムでは、スマートフォン・PC対応のWebアプリケーションとして、地域別タスク管理、タイムスタンプ記録、業務日報、CSV出力、履歴管理を一元提供することで、業務の標準化と効率化を実現する。

### 1.2 目的

本システムの導入により、以下を実現する。

**【現場スタッフ（作業入力者）】**

- 地域別のタスクチェックリストによる業務漏れ防止
- チェック完了時のタイムスタンプ自動記録による作業時間の可視化
- 理想時間対比の遅延アラート（黄色マーカー表示）による品質管理
- 業務日報（新規事案・共有事項）のデジタル入力による報告効率化
- カスタムタスクの追加による柔軟な業務対応
- 入力途中の自動保存（レジューム）によるデータ喪失防止
- 担当者情報の記録による責任の明確化

**【管理者】**

- 全地域の日次進捗の一覧把握と比較
- 過去2ヶ月分の履歴データ参照による傾向分析
- 全体CSVダウンロードによる定量レポート作成
- 業務日報の閲覧による業務品質の把握

**【組織全体】**

- 業務プロセスの標準化と可視化
- 遅延パターンの分析による業務改善
- デジタル化による紙ベース管理の脱却
- データに基づく業務品質の継続的改善

### 1.4 スコープ

本システムは以下の機能を提供する。

**【作業入力画面（機能ID：TK）】**

1. 地域選択機能（担当者選択付き）
2. タスクチェック機能（完了タイムスタンプ記録）
3. 遅延検知・表示機能（理想時間+30分超で黄色マーカー）
4. カスタムタスク追加機能（時間帯・タスク名指定）
5. カスタムタスク削除機能
6. 業務日報入力機能（本日の新規事案 / 業務上の共有事項）
7. 自動保存（レジューム）機能
8. 進捗バー表示機能（地域別完了率）

**【管理画面（機能ID：AD）】**

1. 日付選択機能（過去2ヶ月分の履歴参照）
2. 全体進捗ダッシュボード機能
3. 地域別進捗表示機能（担当者名付き）
4. 全体CSVダウンロード機能
5. 業務日報閲覧機能
6. 古いデータの自動削除機能（2ヶ月超過分）

### 2.2 本システムを利用したサービスイメージ

#### 現場スタッフの一日の流れ（例）

```
07:30  アプリにアクセス → 担当地域（例：関東）を選択、担当者名を登録
       → 7:30タスクから順にチェック開始
       ・e-winds csvダウンロード          → 完了チェック（07:32記録）
       ・e-winds 欠品対応リスト作成        → 完了チェック（07:45記録）
       ・→ line共有                       → 完了チェック（07:48記録）
       ・→ googleスペース格納              → 完了チェック（07:50記録）

08:00  8:00タスクを遂行
       ・セットサブリスト作成              → 完了チェック（08:05記録）
       ・欠品振替業務 → リスト作成 → ...   → 各サブタスクを順次チェック

09:30  9:30タスクを遂行
       ・出荷確定メール送信               → 完了チェック（09:35記録）
       ・ダブルチェック                    → 完了チェック（10:15記録）
         → 30分超過のため黄色マーカー表示

11:40  11:40タスクを遂行

12:00  確定データ保存 → 完了チェック
       ※13時を過ぎた場合は遅延連絡が必要（備考に記載）

13:00  京樽FLI欠品報告、欠品振替サマリ更新 → 完了チェック
       補充業務開始

15:00  フラグ操作（FCL）、フラグ操作（立て替え） → 完了チェック

16:30  SKUフラグチェックシート更新 → 完了チェック

随時   業務日報セクションで新規事案・共有事項を入力
       （例：「おもちゃ：本部商品の為、メーカーから直送不可」）
       
全タスク完了 → 地域選択画面の進捗バーが100%に到達
```

#### 管理者のワークフロー

```
夕方   管理画面にアクセス
       → 本日の日付が自動選択された状態で全地域の進捗を確認
       → 遅延があった地域を特定
       → 「全体CSVダウンロード」でレポート取得
       → 業務日報セクションで各地域の新規事案・共有事項を確認

翌日以降
       → 日付プルダウンから過去日付を選択して過去実績を参照
       → 遅延傾向の分析・業務改善に活用
```

### 2.3 非機能要件

**可用性：**

- 本システムはクライアントサイドWebアプリケーション（Next.js）であり、Vercelにホスティングされる。Vercelプラットフォームの可用性（99.99%SLA）に準拠する
- データ保存はブラウザのlocalStorageを使用するため、サーバー障害時もローカルデータは影響を受けない
- オフライン時も保存済みデータの閲覧は可能（ただし新規保存はブラウザ再アクセスまで遅延する場合がある）

**性能・拡張性：**

- 画面の初回レスポンスタイムは2秒以内を目安とする
- タスクチェック操作のレスポンスは即座（100ms以内）とする
- localStorageの容量制限（一般的に5-10MB）の範囲内で2ヶ月分のデータを保持する
- 将来的にデータベース連携（Supabase / Neon等）への移行を想定した設計とする

**運用・保守性：**

- 2ヶ月超過した古いデータはアプリ起動時に自動削除し、ストレージ容量を管理する
- タスクマスタデータ（`lib/task-data.ts`）の修正により、タスク項目の追加・変更・削除が可能
- 担当者マスタデータの修正により、担当者リストの変更が可能
- レスポンシブデザインにより、スマートフォン・タブレット・PCで利用可能

**セキュリティ：**

- HTTPS（SSL/TLS）通信のみを使用する（Vercelによる自動設定）
- 業務データはlocalStorage（ブラウザローカル）に保存され、サーバーにアップロードされない
- 個人情報は担当者名（氏名のみ）に限定される
- 将来的にデータベース移行時は、認証機能およびアクセス制御の実装を予定する

**バックアップ：**

- localStorageのデータは端末のブラウザに依存するため、CSVダウンロード機能を日次のバックアップ手段として位置づける
- 将来的にデータベース移行時は、日次自動バックアップを実装予定
